# Как добавить маркер

Все маркеры живут в `markers.yaml`. Это единый источник истины: одну запись читают и человек (через `references/*.md`), и линтер (`scripts/stopslop.py`). Добавил запись с полем `regex` — маркер сразу ловится проверкой.

## Поля записи

| Поле | Обязательно | Что в нём |
|---|---|---|
| `id` | да | короткий slug в kebab-case, уникальный по файлу (`em-dash-overuse`) |
| `category` | да | одна из: `phrases`, `structures`, `punctuation`, `formatting`, `morphology` |
| `severity` | да | `high`, `medium`, `low` — насколько надёжно маркер выдаёт машину |
| `title` | да | человекочитаемое имя |
| `why` | да | чем выдаёт ИИ или почему это слабое письмо |
| `bad` | да | список примеров-нарушителей (идут в справочник и в тесты) |
| `fix` | да | что писать вместо |
| `regex` | нет | шаблон Python `re`; без него маркер живёт в справочнике, но не ловится автоматом |
| `flags` | нет | флаги `re`; `i` (IGNORECASE) включён по умолчанию |

## Порядок

1. Добавь запись в нужную секцию `markers.yaml`. Держи `id` уникальным.
2. Если маркер ловится регуляркой, добавь `regex` и проверь его на реальном тексте: `python3 scripts/stopslop.py text.md --id <твой-id>`.
3. Дай 2–4 живых примера в `bad` и точный `fix`.
4. Подними версию корпуса и опиши изменение в `CHANGELOG.md`.

## Проверка регулярки

Линтер компилирует `regex` с флагами `IGNORECASE | MULTILINE`. Если шаблон не скомпилируется, линтер не упадёт, а напишет предупреждение в stderr и пропустит маркер. Прогоняй новый маркер на тексте, где он точно есть, и на тексте, где его быть не должно, чтобы не словить ложные срабатывания.
